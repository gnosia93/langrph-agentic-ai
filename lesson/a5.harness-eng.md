```
Harness Engineering]  ← 시스템/런타임 설계
        ↓
[Context Engineering]  ← LLM에 무엇을 보여줄 것인가
        ↓
[Prompt Engineering]   ← 어떻게 말할 것인가
        ↓
     [LLM]
```

## Prompt Engineering ##
LLM에게 어떻게 지시문을 작성할 것인가 ? 가장 먼저 등장한 개념이고, 가장 좁은 범위이다. 같은 모델이라도 프롬프트에 따라 결과가 다르게 나온다.
LLM 의 응답을 원하는 방향으로 이끌어 내기 위해서 아래와 같은 방식으로 프롬프트를 작성한다. 

* 역할 부여: "너는 Python 전문가야"
* Few-shot: 예시 몇 개 주고 패턴 학습
* Chain of Thought: "단계별로 생각해봐"
* 구조화 출력: JSON, XML 태그로 포맷 지정
* 명확한 제약: "3문장 이내로", "코드만 출력"

[프롬프트 예시]
```
너는 시니어 백엔드 엔지니어야.
다음 코드를 리뷰하고 JSON으로 출력해.

형식:
{
  "issues": [{"line": number, "severity": "high|medium|low", "message": string}]
}

코드:
{{code}}
```


## Context Engineering ##
LLM에게 무엇을 보여줄 것인가. 어떤 정보를 컨텍스트 윈도우에 넣을 것인가? 에 집중하는 방식이다.

### 왜 중요해졌나 ###
* LLM은 컨텍스트 윈도우 안의 정보로만 추론함
* 윈도우가 커졌다고 다 때려넣으면 → 비용↑, 속도↓, 오히려 품질↓ (lost in the middle)
* 필요한 정보를 적시에 적절한 양만 주는 게 기술

### 다루는 것들 ###
* RAG: 필요한 문서만 검색해서 컨텍스트에 주입
* Memory: 대화 이력 관리 (요약, 선별)
* Tool results: 도구 실행 결과를 어떻게 포함할지
* 상태 관리: 에이전트가 지금까지 뭘 했는지
* Context compression: 길어진 컨텍스트를 요약/압축
* Context windowing: 오래된 정보 잘라내기

[context 예시]
```
사용자 질문: "우리 결제 시스템에서 타임아웃 에러 어떻게 처리해?"

❌ 나쁜 컨텍스트: 전체 코드베이스를 다 넣음
✅ 좋은 컨텍스트:
  - 결제 관련 파일 3개 (RAG로 검색)
  - 기존 에러 핸들링 컨벤션 (steering 문서)
  - 최근 관련 PR (선별)
  - 시스템 아키텍처 요약 (compressed)
```
#### 핵심 질문: ####
* 무엇을 포함할 것인가?
* 무엇을 제외할 것인가?
* 어떤 순서로 배치할 것인가?
* 얼마나 압축할 것인가?
* 언제 갱신/교체할 것인가?

## Harness Engineering ##
LLM을 감싸는 시스템 전체를 어떻게 설계할 것인가? 가장 최근에 정립된 개념으로 Claude Code, Cursor, Kiro 같은 에이전트 IDE가 나오면서 주목받기 시작했다.
Harness는 "마구" "골격" 같은 뜻인데, LLM을 실제 작업을 수행하는 시스템으로 감싸는 전체 구조를 말한다.

* Tool 설계: 어떤 도구를 줄 것인가, 도구 인터페이스는 어떻게 생겼는가
* Loop 구조: ReAct, Plan-Execute, 다중 에이전트
* Error handling: LLM이 실수했을 때 어떻게 복구할 것인가
* Validation: 출력 검증, 재시도 정책
* Observability: 로깅, 추적, 디버깅
* Human-in-the-loop: 언제 사람에게 물어볼 것인가
* Safety / Guardrails: 위험한 동작 차단
* Cost / Latency: 모델 선택, 캐싱, 병렬화

#### Claude Code가 코드를 수정하는 흐름 ####
```
1. 사용자 요청 접수
2. [Harness] 관련 파일 찾기 (tool: grep, read)
3. [Context] 찾은 파일을 컨텍스트에 주입
4. [Prompt] "이 코드를 이렇게 수정해줘" 전달
5. LLM 응답 받음
6. [Harness] 응답을 파싱, tool call 실행
7. [Harness] 결과 검증, 실패 시 재시도
8. [Harness] 사용자에게 승인 요청 (위험한 작업)
9. 반영
```
