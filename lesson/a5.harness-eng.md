## Prompt Engineering ##
LLM에게 어떻게 지시문을 작성할 것인가 ? 가장 먼저 등장한 개념이고, 가장 좁은 범위이다. 같은 모델이라도 프롬프트에 따라 결과가 다르게 나온다.
LLM 의 응답을 원하는 방향으로 이끌어 내기 위해서 아래와 같은 방식으로 프롬프트를 작성한다. 

* 역할 부여: "너는 Python 전문가야"
* Few-shot: 예시 몇 개 주고 패턴 학습
* Chain of Thought: "단계별로 생각해봐"
* 구조화 출력: JSON, XML 태그로 포맷 지정
* 명확한 제약: "3문장 이내로", "코드만 출력"

## Context Engineering ##
LLM에게 무엇을 보여줄 것인가. 어떤 정보를 컨텍스트 윈도우에 넣을 것인가? 에 집중하는 방식이다.

### 왜 중요해졌나 ###
* LLM은 컨텍스트 윈도우 안의 정보로만 추론함
* 윈도우가 커졌다고 다 때려넣으면 → 비용↑, 속도↓, 오히려 품질↓ (lost in the middle)
* 필요한 정보를 적시에 적절한 양만 주는 게 기술

### 다루는 것들 ###
* RAG: 필요한 문서만 검색해서 컨텍스트에 주입
* Memory: 대화 이력 관리 (요약, 선별)
* Tool results: 도구 실행 결과를 어떻게 포함할지
* 상태 관리: 에이전트가 지금까지 뭘 했는지
* Context compression: 길어진 컨텍스트를 요약/압축
* Context windowing: 오래된 정보 잘라내기

[context 예시]
```
사용자 질문: "우리 결제 시스템에서 타임아웃 에러 어떻게 처리해?"

❌ 나쁜 컨텍스트: 전체 코드베이스를 다 넣음
✅ 좋은 컨텍스트:
  - 결제 관련 파일 3개 (RAG로 검색)
  - 기존 에러 핸들링 컨벤션 (steering 문서)
  - 최근 관련 PR (선별)
  - 시스템 아키텍처 요약 (compressed)
```

* 무엇을 제외할 것인가?
* 어떤 순서로 배치할 것인가?
* 얼마나 압축할 것인가?
